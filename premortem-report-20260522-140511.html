OpenTaint Playground — Plan 2: the live re-run backend

Frame: it's 6 months out. The live backend launched and failed. 9 investigators worked backward to explain how. Generated 2026-05-22.

Synthesis

Most likely failure
The "warm pool" is a fiction → uniformly slow & expensive.
Autoscale-to-zero and ephemeral per-job containers cannot hold a hot JVM/Gradle daemon or an in-memory compiled model across requests, so nearly every Run pays full warmup + from-scratch compile. The "rule-only edits are fast / code edits are incremental" premise never materializes — directly causing the latency (#3) and cost (#2) failures. It's not an edge case; it's the default behavior of the chosen architecture.
Most dangerous failure
Compiling attacker-controlled Java → container escape.
"Edit existing files only" doesn't confine code execution: the Gradle/Java build runs code at compile time. A determined attacker gets exec in the worker and escapes via an unpatched runtime — a security company's infra used as a cryptominer/pivot, secrets exfiltrated via DNS despite egress-deny. Lower probability than the perf issues, but existential reputational damage.
The hidden assumption
That interactivity requires running the full, unmodified heavyweight engine live, per request, on arbitrary edits. opentaint is a minutes-long, multi-GB, whole-program JVM batch tool that compiles arbitrary code — treating "Run scan" as a cheap web call is the root from which cost, latency, abuse, ops, and security failures all grow. Most of them dissolve if the live primitive is reshaped.
    The revised plan
  1. Don't run arbitrary code live. Lead with rule-editing. Persist opentaint's compiled base model as a build artifact in the image; "rule-only" Runs load that artifact + apply edited rules + re-run analysis — no Gradle, no attacker code, no JVM-daemon dependency. This is the safe, cheap, genuinely-fast live capability. (Kills #1 surface, most of #2/#8; fixes the #5 fiction.) Verify opentaint supports compile→persisted-model→scan with custom rules.
  2. For code edits: precompute, don't live-compile. Offer a small curated set of "what-if" code variants with PRE-COMPUTED results, or gate live code-compile behind a session challenge + a hard global daily spend cap that auto-disables the live path (graceful fallback to the static result + Install-CLI CTA). (#2, #4, #8)
  3. Pin the engine by digest; prove parity. Pin opentaint@sha256:… (never :latest); regenerate the static read-path SARIF from the SAME digest; CI asserts a no-edit live run == the static baseline (finding count + paths). (#6)
  4. Anchor-based line mapping. Run the engine on the EXACT editor bytes (no server-side reformatting); map taint steps to content anchors (token/line-hash), not absolute lines; one tested 0/1-based converter. (#7)
  5. Treat the worker as hostile. microVM isolation (gVisor/Firecracker), patched host pool, deny ALL egress incl. DNS, no tokens/secrets in the image, one tenant per microVM, ephemeral. (#1)
  6. Bound cost & abuse hard. Global concurrency + hard daily/monthly spend ceiling that disables live path; per-SESSION challenge tokens (defeats IP rotation); analysis time/mem caps tuned so pathological inputs fail fast; queue-position UI with graceful fallback. (#2, #4, #8)
  7. Name an operational owner before building — or don't ship it. On-call owner + budget alerts + a synthetic monitor exercising the FULL live path + automated image CVE rebuild. No owner → keep the static playground + "run it locally" CTA. (#9)
Pre-launch checklist
  • Load test: p95 time-to-result < 60s at 3 concurrent AND cost-per-run < $0.15 — else don't launch the live path.
  • Security: worker in a microVM, egress fully denied incl. DNS, no tokens in image, host pool patched (runc CVE), and a red-team file-edit RCE attempt fails.
  • Version parity gate: opentaint pinned by digest; no-edit live run == static baseline; static SARIF regenerated from that digest.
  • Hard global daily spend cap wired to auto-disable → graceful fallback; per-session challenge live; analysis timeout/mem tuned for pathological inputs.
  • Named ops owner + full-path synthetic monitor + image CVE auto-rebuild; pre-warm the pool before launch (never launch from zero).
9 failure investigations

Sandbox / RCE & container escape#1

Likelihood: MedDamage: Critical

A researcher posts a PoC: an edit to an existing .java file runs a payload during Gradle compilation. Egress-deny blocks TCP, but DNS exfil (UDP/53 to the internal resolver) leaks the image's registry token. A second actor runs a cryptominer to the wall-clock limit. Week five: a worker on an unpatched runc node escapes to the host — a security company's infra becomes a pivot, and it makes the rounds before anyone notices.

Underlying assumption

Confining which files can be edited equals confining what code executes — but the Java/Gradle build is itself a Turing-complete execution environment driven by file content.

Early warning signs

• Anomalous high-entropy DNS lookups in worker build logs.
• "Compile-only" jobs showing CPU that outlasts javac exit.

Unbounded organic cost#2

Likelihood: HighDamage: High

800 visitors, half click Run — 90–180s CPU and 6–10GB RAM each. A newsletter mention drives 25 concurrent jobs; the autoscaler obeys (that's the problem). One 24h window burns $600 against a $300/month budget. The team emergency-caps concurrency to 2; queue depth hits 80, median wait 22 min; users leave. By month four the live path is a spinner that never resolves and is pulled from the landing page.

Underlying assumption

"Autoscale-to-zero controls cost" — ignoring that ordinary success means jobs always run, and each is expensive enough that moderate organic traffic exceeds a small team's budget.

Early warning signs

• Cost-per-run > $0.15 in load tests.
• Week-one cost-per-unique-visitor > $0.50.

Latency / p95 kills conversion#3

Likelihood: HighDamage: High

Solo testing looks fine (rule edit 8s, code edit 45s), so it ships. Real traffic fills the one-per-slot queue; a 45s compile becomes a 3-minute wall-clock wait; p95 > 4 min. The "Resolving deps… Compiling…" theatre burns its goodwill in 90 seconds, then reads as a broken spinner. Edit→run conversion falls from ~40% solo to 8%; returning users avoid the live path entirely.

Underlying assumption

Warm containers cut latency enough that queue wait + compile stays within visitor patience — but they remove only one of three compounding delays.

Early warning signs

• p95 > 60s at just 3 concurrent scans in load test.
• Edit→run conversion < 25% week one; abandons at the 60–90s mark.

Launch-spike queue starvation#4

Likelihood: HighDamage: High (one-shot)

The HN post hits at 9:47am onto a system idle overnight — every container scaled to zero. The first 50 Runs land while cold starts take 90–120s each. The queue hits 200+, per-IP limits start 429-ing eager first-timers, and the comment thread fills with "just spun for me." 4,000 visitors, 94% bounce on the scan page, 11 installs. The one-shot launch window is gone.

Underlying assumption

Autoscale-to-zero and launch timing are independent — so no explicit pre-warm step is needed before a scheduled public launch.

Early warning signs

• Median time-to-first-result > 90s from a cold (zero-container) start.
• Queue depth > 20 within 60s of a simulated burst with nothing warm.

The "warm pool" is a fiction#5

Likelihood: Very HighDamage: High

Scale-to-zero destroys the JVM process and in-memory compiled model on every idle event; ephemeral per-job containers never serve a second build while warm, so the Gradle daemon is born and dies in one 90s window. The "rule-only reuses the compiled model" fast path is equally fictional — the model lived in the previous container's heap. Every job recompiles from scratch. To hit acceptable p50 they must keep one instance always-on, erasing the scale-to-zero savings.

Underlying assumption

"Warm" JVM/daemon/model state persists across jobs — conflating pool-level readiness (images pre-pulled) with process-level warmth (live heap, active daemon), which die at teardown.

Early warning signs

• p95 > 3× p50 once >10 min elapses between runs.
• Gradle daemon reuse rate across container boundaries = 0 in load tests.

Read-path vs live-path result drift#6

Likelihood: HighDamage: High (credibility)

An AppSec engineer loads the instant view (14 findings), clicks Run with no edits, and gets 11 — two paths gone, one line shifted. Cause: the static SARIF was built at opentaint a3f8c1 / demo v1.0; :latest is now b9d44e with changed rule defaults, and a README commit shifted a line. No version pin, no badge, no diff. She tweets "claims deterministic, gives different findings twice — hard pass." It contradicts opentaint's headline claim with exactly the audience the funnel needs.

Underlying assumption

opentaint:latest at request time produces identical findings to the build-time snapshot — treating "deterministic" as a property of the algorithm, not of (algorithm + version + inputs).

Early warning signs

• Deployed :latest digest ≠ the digest used for the static SARIF.
• Post-deploy live run of the unedited demo returns finding count ≠ baseline.

Edit → SARIF line-mapping bugs#7

Likelihood: Med-HighDamage: High (centerpiece)

The user adds a 5-line docblock; the sink moves to line 47 but SARIF still says 42 and the decoration paints a comment line. The pipeline normalizes source (formatter/AST canonicalization) before analysis, so SARIF lines map to the normalized artifact, not the editor buffer. Monaco is 0-indexed, SARIF 1-indexed — a silent off-by-one. The corrosive case: 80% land right, 20% (a reformatted transitive file) are wrong; the engineer trusts the 80% and misses a real sink.

Underlying assumption

SARIF line numbers map 1:1 to the raw Monaco buffer, with no transformation, normalization, or indexing difference.

Early warning signs

• Decorations land a constant N lines off on a fresh, no-edit scan.
• Decoration accuracy degrades with lines-added (positional, not anchor-based mapping).

Malicious abuse / denial-of-wallet#8

Likelihood: Med-HighDamage: High

A botnet on rotating residential proxies hammers /scan with crafted deep call-graphs that force exponential dataflow state explosion — every job hits the 5-min timeout, permanently saturating the pool; legit users see queue-full 100% of the time. The 10 req/min per-IP cap, split across 500 nodes, still floods 2,500 pathological jobs/min. Spend goes $300→$4,000 in one cycle. The team kills it after the second invoice; no affordable mitigation (PoW hurts mobile, CAPTCHA adds a dep, tighter timeouts just make attacks cheaper).

Underlying assumption

Per-IP rate limiting provides meaningful abuse resistance when cost-per-request is high enough to attract adversaries with botnet access.

Early warning signs

• p95 job duration > 90% of the wall-clock timeout within 48h of launch.
• Active containers pinned at ceiling while unique sessions stay flat.

No operational owner#9

Likelihood: HighDamage: High (slow rot)

Launch excitement fades; the engine team enters a sprint. A new :latest changes a JVM flag and breaks the startup probe — containers never pass health checks, the autoscaler spins up more, costs double, and it's dead for 11 days because the monitor only checks the gateway. A dependency CVE sits unassigned 3 weeks until a researcher tweets it; the rushed patch ships a resource-limit regression that OOM-kills scans >50 lines for another month. By month six ~30% of requests error and the team debates shutting it down — signaling Seqra can't operate production software.

Underlying assumption

Infrastructure that needs continuous operational attention will get it from a team whose mandate and attention are elsewhere (the engine/CLI).

Early warning signs

• Time-to-patch a disclosed worker-image CVE > 7 days.
• Gateway returns 200 but p95 scan latency > 8s for 48h+ with no incident ticket.
